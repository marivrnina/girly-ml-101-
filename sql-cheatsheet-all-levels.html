Level 1
Foundational SQL

The queries you reach for every single day — filtering, sorting, grouping, and joining tables together.

01 / 03 — start here

SELECT · WHERE · ORDER BY

What it is: the core sentence structure of SQL — pick your columns, narrow to matching rows, set the sequence.

Used for: nearly every query you'll ever write starts here.

SELECT name, revenue
FROM sales
WHERE revenue > 1000
ORDER BY revenue DESC;

GROUP BY · HAVING

What it is: GROUP BY collapses rows into buckets by shared values; HAVING filters those buckets after aggregation (WHERE can't do this).

Used for: "total sales per region," "count of orders per customer," any per-group summary.

SELECT region, SUM(revenue)
FROM sales
GROUP BY region
HAVING SUM(revenue) > 5000;

Aggregate Functions

What it is: functions that collapse many rows into a single summary number.

Used for: KPIs and dashboard numbers — totals, averages, counts, extremes.

COUNT(*)   SUM(col)
AVG(col)    MIN(col)
MAX(col)

Filtering Shortcuts

What it is: pattern and range matching that replaces long chains of OR conditions.

Used for: searching text patterns, checking membership in a list, filtering numeric/date ranges.

name LIKE 'A%'
status IN ('active','trial')
price BETWEEN 10 AND 50

Joins

What it is: a way to combine rows from two tables based on a related column (usually an ID). Real data lives in separate tables, so joins stitch it back together.

Used for: connecting orders to customers, transactions to products — any relational lookup. This is the one to have muscle memory for.

SELECT o.order_id, c.customer_name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.customer_id;

LEFT JOIN   → keep all rows from the left table
RIGHT JOIN  → keep all rows from the right table
FULL JOIN   → keep everything from both

02 / 03 — leveling up

Subqueries

What it is: a query nested inside another query. Non-correlated subqueries run once and return a fixed value/set; correlated ones re-run for every outer row and can reference it.

Used for: comparing a row against an aggregate ("above-average salary"), or checking existence against another table.

SELECT name FROM employees
WHERE salary > (
  SELECT AVG(salary) FROM employees
);

CASE Statements

What it is: conditional branching evaluated top-to-bottom, stopping at the first true condition — SQL's version of if/elif/else.

Used for: bucketing values into tiers/categories, recoding messy data, building flags.

SELECT order_id,
  CASE
    WHEN total > 100 THEN 'high'
    WHEN total > 20  THEN 'mid'
    ELSE 'low'
  END AS tier
FROM orders;

Set Operations

What it is: operations that treat two query results as sets and combine or compare them — both queries need matching column counts/types.

Used for: merging similar data from different tables, finding overlaps or differences between two lists.

query_a
UNION       -- combine, no dupes
INTERSECT   -- rows in both
EXCEPT      -- rows only in first
query_b;

Date & String Functions

What it is: built-in functions that transform raw text and timestamps into usable, analysis-ready values.

Used for: rolling dates up to month/year, calculating durations, cleaning inconsistent names/emails.

DATE_TRUNC('month', order_date)
DATEDIFF(end_dt, start_dt)
CONCAT(first, ' ', last)
SUBSTRING(email, 1, 5)

CTEs — the readability upgrade

What it is: a temporary, named result set defined with WITH, that exists only for the query that follows it.

Used for: breaking complex logic into readable, top-to-bottom steps instead of deeply nested subqueries.

WITH monthly_sales AS (
  SELECT customer_id, SUM(amount) AS total
  FROM orders
  GROUP BY customer_id
)
SELECT * FROM monthly_sales WHERE total > 500;

The queries that separate "can write SQL" from "thinks in SQL" — window logic, chained transformations, and knowing why a query is slow.

03 / 03 — advanced

Ranking Windows

What it is: a window function assigns a rank/number to each row within a PARTITION, without collapsing rows like GROUP BY does — you keep every row and add a value alongside it.

Used for: top-N per group, deduplication, leaderboard-style rankings.

SELECT name, dept,
  ROW_NUMBER() OVER (
    PARTITION BY dept
    ORDER BY salary DESC
  ) AS rnk
FROM employees;

LAG / LEAD

What it is: window functions that reach into a neighboring row (previous with LAG, next with LEAD) within an ordered partition, without a self-join.

Used for: month-over-month change, detecting sequence gaps, comparing consecutive events.

SELECT month, revenue,
  revenue - LAG(revenue) OVER (
    ORDER BY month
  ) AS mom_change
FROM monthly_revenue;

Running Totals

What it is: an aggregate applied over a defined "frame" of rows (e.g. all rows up to the current one) rather than the whole partition at once.

Used for: cumulative sums, moving averages, progress-to-date metrics.

SELECT order_date,
  SUM(amount) OVER (
    ORDER BY order_date
    ROWS BETWEEN UNBOUNDED PRECEDING
    AND CURRENT ROW
  ) AS running_total
FROM orders;

Dialect Quirks to Know

What it is: SQL isn't fully standardized — each engine (MySQL, Postgres, etc.) enforces certain rules differently, especially around aggregation.

Used for: knowing why a query that works in one engine throws an error in another.

-- MySQL: no LIMIT inside IN (...)
-- ONLY_FULL_GROUP_BY: every
   non-aggregated column must
   appear in GROUP BY

Chained CTEs

What it is: multiple CTEs defined in sequence, where each one can reference the CTE before it — effectively a mini data pipeline inside one query.

Used for: multi-step transformations (aggregate → rank → filter) that would otherwise need nested subqueries or a temp table.

WITH step1 AS (
  SELECT customer_id, SUM(amount) AS total
  FROM orders GROUP BY customer_id
),
step2 AS (
  SELECT *,
    NTILE(4) OVER (ORDER BY total DESC) AS quartile
  FROM step1
)
SELECT * FROM step2 WHERE quartile = 1;
